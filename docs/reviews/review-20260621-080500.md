## 1. Executive summary

1. **BUG — Chained set operations are compiled without parentheses, so the SQL does not mean what the algebra tree says.** `a.difference(b.difference(c))` renders the *correct* algebra `(a − (b − c))` but emits `… EXCEPT … EXCEPT …`, which SQLite evaluates left-to-right as `(a − b) − c`. Reproduced: returns `{1}` where the tree means `{1, 3}`. Mixed `UNION`/`INTERSECT` chains are additionally wrong on standard backends (Postgres) where `INTERSECT` binds tighter. This breaks the one promise the library exists to keep: algebra ⇔ SQL fidelity. (`compiler.py:235`)
2. **BUG — A typo'd aggregate column silently returns garbage instead of erroring.** `Grouping.__post_init__` validates group *keys* but never the *aggregate* attributes, and `_compile_grouping` builds `SUM("nope")` without consulting the schema. On SQLite the unknown quoted identifier degrades to a string literal, so `sp.group("sno", bad=sum_("nope")).collect()` returns `[('S1', 0.0), …]` — no error. The same node's `.schema()` *does* raise `AttributeError_`, so `print()` and `.collect()` disagree. (`relation.py:794`, `compiler.py:465`)
3. **SMELL — Every expression-tree node has value-equality enabled, and value-equality raises.** The node dataclasses use the default `eq=True`; comparing two nodes invokes the generated `__eq__`, which eventually calls `bool()` on an `Attr` comparison and raises `PredicateError`. `node1 == node2`, `node in [...]`, and `set(nodes)` all blow up. The same `eq=True`+`frozen=True` makes `Grouping`/`Rename` unhashable (dict fields), so "frozen" nodes can't go in a set at all. (`relation.py:335` and every node class)
4. **DESIGN — `Predicate.sql()` / `CompoundPredicate.sql()` / `NotPredicate.sql()` are dead, dialect-blind duplicates of the real compiler.** Nothing outside the predicate hierarchy calls them; they hardcode `?` placeholders and emit unqualified column names, contradicting the "all SQL flows through the Dialect" invariant the rest of the codebase upholds. A maintainer who reaches for `pred.sql()` gets qmark-only, join-unsafe SQL. (`predicates.py:175`, `predicates.py:231`, `predicates.py:272`)
5. **DESIGN — The "eager validation at construction" invariant, asserted in four module headers and the test suite, has two holes.** Aggregate-attribute existence (finding 2) and `Equijoin` output-collision both surface only when `_schema()` is later called, not at construction. (`relation.py:616`, `relation.py:794`)

The good news up front: **the author resolved every finding from the April review** — column-order desync, `AttributeError_` inheritance, division aliasing, the PRAGMA injection, bag-mode threading, CI (ruff + mypy + 3.10–3.13 matrix + a real Postgres job), the type-prefix boundary, cursor closing, and the docs. ruff, mypy, and the 87-test suite are all green locally. The findings below are new ground, mostly in the set-operator and aggregate corners that the prior pass did not probe.

## 3. Findings by category

### Correctness

- **BUG — `compiler.py:235` (`_compile_setop`) emits chained set operations with no parentheses; SQL grouping diverges from the algebra tree.**
  `_compile_setop` returns `f"{left_sql} {keyword} {right_sql}"`, and `_visit` of a nested `_SetOp` returns a bare `"… UNION …"` string. So a tree whose shape encodes grouping is flattened. Two distinct failure modes, both verified:
  - *Right-nested `EXCEPT`/`UNION` — wrong on SQLite too.* `a.difference(b.difference(c))` with `a={1,2,3}`, `b={2,3,4}`, `c={3,4,5}`:
    ```
    algebra : (a − (b − c))            # display is CORRECT
    SQL     : … FROM "a" EXCEPT … "b" EXCEPT … "c"
    result  : {1}                      # SQLite left-assoc = (a−b)−c
    correct : {1, 3}                   # a − (b − c)
    ```
  - *Mixed `UNION`/`INTERSECT` — wrong on standard backends.* `a.union(b).intersect(c)` emits `… UNION … INTERSECT …`. SQLite evaluates strictly left-to-right (so it happens to match), but the SQL standard — and PostgreSQL, which the package advertises — gives `INTERSECT` higher precedence, yielding `a UNION (b INTERSECT c)`. The library's own `[postgres]` extra and README examples make this a live target.

  This is the single most important finding: the entire pedagogical premise is that the rendered algebra and the executed SQL are the same query, and here they are provably not. Note the compiler author already knows parentheses are load-bearing for `EXCEPT` — `_compile_division` carefully parenthesizes its inner `EXCEPT` — so this is an oversight in one code path, not a misunderstanding.
  *Fix:* when a set-op operand is itself a `_SetOp`, wrap it — either parenthesize, or route it through subquery wrapping the way `_as_source` already does for non-`Relation` nodes (`SELECT * FROM (<compound>) AS tN`), which is also the most portable form across SQLite/Postgres.

- **BUG — `relation.py:794` / `compiler.py:465` — a non-existent aggregate attribute is never validated and silently yields wrong data.**
  `Grouping.__post_init__` checks `self.keys` against the child schema but not the attributes inside `self.aggs`. `_compile_grouping` then emits `f"{agg.func}({attr})"` straight from `agg.attr` without touching `_schema()`. Verified: `sp.group("sno", bad=sum_("nope")).collect()` → `[('S1', 0.0), ('S2', 0.0)]`. SQLite's double-quoted-identifier-as-string-literal misfeature turns `SUM("nope")` into `SUM('nope') = 0.0`; no error is raised. The *same* node's `.schema()` raises `AttributeError_` (via `AggSpec.output_domain → schema[self.attr]`), so `print(node)` and `node.collect()` give contradictory outcomes on identical input.
  *Fix:* in `Grouping.__post_init__`, validate each `agg.attr != "*"` against the child schema (calling `agg.output_domain(self.child._schema())` would do it and matches the eager pattern used for keys).

- **SMELL — `engine.py:284` — the Postgres introspection branch hardcodes `%s` and relies on driver tolerance.**
  The comment is admirably honest that psycopg reports `pyformat`, not `format`, and that the bare `%s` works only because psycopg forgives it. This is a documented, deliberate shortcut, not a defect — flagged only so the next person who adds a strict-paramstyle PG driver knows to route it through the Dialect like everything else.

### Mutability / API design

- **SMELL — `relation.py:335` and every node class — `@dataclass(frozen=True, repr=False)` leaves `eq=True`, which is both unused and actively harmful.**
  The expression tree wants *identity* semantics (the engine-identity check uses `is`; `Attr.__hash__` deliberately hashes on `id(self.source)`). But the generated structural `__eq__` walks into `Attr.__eq__`, which returns a `Predicate`; tuple comparison then calls `bool()` on it and raises `PredicateError`. Verified: `node1 == node2`, `node1 in [node2]` both raise. Separately, `eq=True`+`frozen=True` synthesizes `__hash__` over the fields, and `Rename.mapping`/`Grouping.aggs` are `dict` → `hash(node)` raises `TypeError: unhashable type: 'dict'`. So these "frozen" nodes are neither comparable nor hashable.
  *Fix:* set `eq=False` on the node dataclasses (one change per class, or factor a shared decorator). They fall back to object identity for both `__eq__` and `__hash__`, which is what the tree actually wants, and it removes the dependency on predicate/dict hashability.

- **DESIGN — `relation.py:616` (`Equijoin.__post_init__`) validates only attribute existence; the output-collision check lives in `_schema()`.**
  Constructing `a.equijoin(b, "k", "j")` where `a` and `b` share a third column does *not* raise at construction (verified); the `SchemaError` appears only when `_schema()` is first evaluated. Every sibling node (`Projection`, `Rename`, `CrossProduct`, `ThetaJoin`, `Division`, `Grouping` keys) validates eagerly in `__post_init__`, and the invariant is asserted in `schema.py:10`, `relation.py:41`, and the test suite's Phase 1 banner. This was noted in the prior review and is unchanged. Combined with finding 2, the eager-validation guarantee has two holes; closing both would let the docs' claim stand literally.

- **SMELL — `OuterJoin.outer_join(..., how="full")` default (`relation.py:147`).** A bare `r.outer_join(s)` silently means full outer join. The README always passes `how=`, and full is a defensible default, but a `KeyError`-free typo like `how="LEFT"` raises `ValueError` (good) while the no-arg call quietly picks the widest, most expensive join. Minor; consider making `how` required for explicitness in a teaching API.

### Docs vs reality

- **SMELL — README "zero additional dependencies … start immediately" understates SQLite version requirements for two advertised operations.** `RIGHT`/`FULL OUTER JOIN` (`compiler.py:421`) require SQLite ≥ 3.39 (2022). The version bundled with Python varies by platform/build; a student on an older system Python will get `sqlite3.OperationalError: RIGHT and FULL OUTER JOINs are not currently supported` from a README-blessed call. (Verified working here on SQLite 3.50.4.) A one-line "outer joins other than `left` need SQLite ≥ 3.39" note would close the gap.
- **NIT — `display.py:486` (`format_sql`) splits `LEFT/RIGHT/FULL OUTER JOIN` across two lines.** Because the keyword list processes `JOIN` before `LEFT OUTER JOIN`, the first pass inserts a newline before the inner `JOIN`, after which the multi-word rule can't match. Output:
  ```
  FROM "L" LEFT OUTER
  JOIN "R" ON L."sno" = R."sno"
  ```
  Cosmetic and non-corrupting (as the docstring warns), but outer-join SQL is itself a teaching artifact, so the mangled rendering undercuts it. Reorder the keyword list longest-first, or special-case the `… OUTER JOIN` forms before bare `JOIN`.

### Security

- **No finding.** The "literals never interpolated" invariant holds on the live path: every value flows through `self.params` + `Dialect.placeholder()`, identifiers go through `quote_identifier` with proper quote-doubling (`engine.py:129`), DDL types come from the `SUPPORTED_DOMAINS` lookup (not free-form), and the previously-vulnerable PRAGMA path is now gated by `isidentifier()`. The only string-built SQL fragments are structural (keywords, aliases, quoted identifiers). The dead `Predicate.sql()` family (finding 4) hardcodes `?` but is unreachable, so it is not a live injection concern — only a maintenance trap.

## 4. Open questions

- **Are compound set expressions in scope?** The README only shows two-operand set ops. If chained/mixed set ops are considered out of scope, finding 1 is "document the limitation and reject `_SetOp` operands of `_SetOp`"; if they're in scope (the algebra section implies free composition — "compose them freely, just like arithmetic"), it's a parenthesization fix. The wrong-vs-displayed-algebra divergence makes me treat it as a bug regardless.
- **Is node value-equality ever wanted?** If some future feature needs structural equality of expression trees (e.g. an optimizer that dedupes subtrees), it cannot rely on `==` while predicates hijack `__eq__`; it would need an explicit structural key. Worth deciding now, because `eq=False` is the clean fix today and reversing it later is harder.
- **Does the Postgres CI job actually round-trip parameters?** I did not run `test_postgres.py` (no PG instance). The non-`qmark` paramstyle branches and the `%s` introspection shortcut are only as trustworthy as that job's assertions.
